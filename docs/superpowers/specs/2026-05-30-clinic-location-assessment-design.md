# 診所選址評估系統 — 設計規格

**日期**：2026-05-30
**地點**：桃園市龜山區樂善二路 503 號（單一固定候選點）
**目標科別**：家醫科、功能醫學、減重診所、精神/身心科、醫美

## 1. 目的與範圍

針對單一固定地點，自動抓取台灣開放資料，對 5 個科別各算出 0–100 的「開業適合度」分數，輸出成可圖示化、可看趨勢的 HTML 網站，並由 n8n 每月/每季排程刷新、累積歷史快照。

**範圍內（MVP）**
- 單一地址的評估（樂善二路 503 號）
- 5 科別各自評分與排名
- 雙商圈半徑分析（步行 1km、車程 3km）
- HTML 網站輸出 + 趨勢圖 + 地圖
- 每月/季自動刷新與歷史快照累積

**範圍外（未來擴充）**
- 多地址 / 全台通用化（之後只需改地址參數）
- 租金的自動抓取（開放資料精度不足，改為手動輸入）

## 2. 系統架構（方案 A）

```
n8n Cloud (排程通知)            macOS launchd (每月觸發)
        ▲                              │
        │ 完成通知                      ▼
        └────────────────  Python 分析核心
                              ├─ 抓取層
                              ├─ 分析層
                              └─ 快照層
                                     │
                                     ▼
                          靜態 HTML 網站 (history.json)
                          Chart.js + Leaflet 圖示化
```

部署方式採 **(i)**：Python 跑在使用者的 Mac，用 launchd 每月自動執行、重新產生網站並推送至靜態主機；n8n 僅負責「完成通知」。

## 3. 元件拆解

各元件單一職責、介面明確、可獨立測試。

### 抓取層 `data_sources/`
- `geocode.py` — TGOS 地址定位 API：候選點與競爭院所地址 → 經緯度（WGS84，注意 TWD97 轉換）。**核心前提，失敗即中止。**
- `nhi_clinics.py` — 健保特約院所 CSV → 過濾科別 → geocode → 半徑內計數。資料含地址與科別、不含座標，需自行 geocode。
- `tdx_transit.py` — TDX 運輸平台 API：捷運站、公車站、停車場點位。需 OAuth2 金鑰（Client ID/Secret），流量上限每秒 50 次。
- `osm_poi.py` — OSM Overpass API：超商、超市、學校等錨點 POI（補官方開放資料缺口）。
- 本地參考庫 `data/reference/`：
  - SEGIS 250m 人口網格（人口數、年齡、性別）— SHP/GeoJSON 下載
  - 財政部村里綜所稅所得統計 — CSV
  - 行政區界圖（村里/鄉鎮）— SHP/GeoJSON
  - 半靜態資料，以獨立指令定期更新（人口月/季、所得年度）

### 分析層 `analysis.py`
- **雙半徑空間統計**：對步行 1km、車程 3km 兩層各自彙總：
  - 人口總量與密度（250m 網格中心落在半徑內者加總）
  - 年齡結構、25–45 歲與女性比例
  - 家戶所得（村里中心落在半徑內者，面積加權近似）
  - 同科別競爭密度（半徑內該科別院所數）
  - 交通可及（半徑內站點/停車場數、最近站距離）
  - 錨點數（超商/超市/學校等）
- **評分引擎**：讀 `config/specialties.yaml` 的權重矩陣與門檻，各因子正規化（min-max 或門檻分段）後加權 → 各科別 0–100 分。純函式、決定性。

### 快照層 `snapshot.py`
- 每次執行附加一筆 `{日期, 各科別分數, 各因子原始值與正規化值}` 至 `history.jsonl`。
- 趨勢即為月度快照之累積。

### 輸出層 `site/`（純靜態，零建置步驟）
- `index.html` + JS 讀 `history.json`
- 圖示：
  1. 五科別雷達圖（本月適合度比較）
  2. 各科別分數折線趨勢圖（跨月）
  3. 因子細項長條圖（依半徑分層）
  4. Leaflet/OSM 地圖：候選點 + 競爭院所 + 雙半徑圈 + 交通點
- 可託管於 GitHub Pages 或任何靜態主機。

### 設定檔 `config/specialties.yaml`
- 各科別對各因子的權重與門檻，可調整不需改程式。
- 初始權重矩陣（⬆ 越多權重越高、⬇ 表競爭為負向）：

| 因子 | 家醫科 | 功能醫學 | 減重 | 精神/身心科 | 醫美 |
|---|---|---|---|---|---|
| 總人口/密度 | 高 | 中 | 中 | 高 | 中 |
| 25–45歲/女性比 | 中 | 中 | 高 | 中 | 高 |
| 家戶所得（自費力） | 中 | 高 | 高 | 中 | 最高 |
| 同科別競爭密度（負向） | 中 | 低 | 高 | 中 | 最高 |
| 交通/停車可及 | 高 | 中 | 中 | 高 | 中 |
| 能見度/臨街 | 中 | 中 | 中 | 中（重隱私） | 高 |

## 4. 資料流（每次刷新）

1. macOS launchd 每月/季觸發 Python 核心。
2. geocode 候選點 → 對每個半徑、每個資料源做空間統計。
3. 套用科別權重 → 算分 → 寫入 history snapshot。
4. 重新產生 `history.json` → 推送靜態網站。
5. 呼叫 n8n webhook → n8n 寄出「本月評估已更新」通知（含重點摘要）。

## 5. 錯誤處理

- 外部 API（TDX/TGOS/Overpass）失敗 → 該因子標記「資料缺漏」、沿用上次快照值並標註，不讓單一來源中斷整份報告。
- geocode 失敗 → 中止並通知（核心前提，不可降級）。
- 租金因子 → 由使用者手動輸入實際洽談租金，工具僅做損益試算參考，不自動抓取。
- 店面實體屬性（能見度/臨街、樓層、面寬、坪數、有無電梯）→ 無開放資料來源，由使用者手動輸入，併入評分引擎。

## 6. 測試策略

- **評分引擎**：固定 fixture 餵已知因子值，驗證各科別分數可重現。
- **半徑彙總**：合成點位驗證距離篩選與面積加權正確。
- **抓取器**：mock API 回應，不打真實外部 API。

## 7. 技術選型

- Python：`requests`、`pandas`、`shapely`（單點與點位用 haversine 距離，不需 geopandas 重相依）。
- 設定：`specialties.yaml`（權重可調）。
- 前端：Chart.js（圖表）+ Leaflet/OSM（地圖），純靜態。
- 排程：macOS launchd。
- 通知：n8n Cloud webhook。
